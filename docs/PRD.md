# PRD — Persona Mirror (Client-First React 재설계)

> 문서 버전: 1.1 · 작성일: 2026-05-31 · 상태: React UI/UX 동기화 반영

## 1. 배경 및 목적

Persona Mirror는 대화 기록으로 상대방의 페르소나·말투를 분석하고, 받은 메시지에서 **"상대방이 원하는 답변"** 을 추론해주는 모바일 웹 앱이다.

기존 버전은 **Python(FastAPI) + 로컬 Ollama + 서버 디스크 JSON 저장** 구조였다. 이번 재설계의 목적은 다음과 같다.

1. **런타임을 Node + React 기반으로 전환** — 배포·운영 생태계를 JS/TS로 통일하고, 형제 앱 `say-awsomely`와 같은 UI 개발 체계를 사용.
2. **개인정보를 서버에 저장하지 않음** — 페르소나·대화 기록·분석 결과를 모두 **사용자 브라우저**에 보관.
3. **사용자 소유 LLM 키 사용** — 첫 실행 시 Google AI Studio(Gemini) API 키를 입력받아 **쿠키**에 저장하고, 그 키로 LLM을 호출.
4. **서버 부담 최소화** — 서버는 정적 파일만 서빙. LLM 호출·데이터 저장은 전부 클라이언트가 담당.

핵심 한 줄: **"개인정보는 개인 브라우저에, 연산은 사용자 키로, 서버는 그저 정적 호스트."**

## 2. 목표(Goals) / 비목표(Non-Goals)

### Goals
- G1. 3탭 모바일 SPA(페르소나 / 분석하기 / 기록)를 유지하되, 상태바·하단바·모달·토스트·애니메이션은 **React 형제 앱 `say-awsomely`와 동일한 UX 패턴**으로 동기화한다.
- G2. 페르소나·분석 기록은 **IndexedDB**에 저장하고, 서버로 전송하지 않는다.
- G3. 첫 진입 시 Gemini API 키를 입력받아 **쿠키**에 저장하고, 미입력 상태에서는 분석 기능을 막는다.
- G4. 브라우저가 **Gemini API를 직접 호출**한다. (대화 데이터가 우리 서버를 경유하지 않음)
- G5. Node 서버는 **빌드된 정적 자산만 서빙**한다(필요 시 SPA fallback). API/DB/세션 없음.
- G6. 프로덕션 수준의 소스 구조(타입 안전, 모듈 분리, 빌드 파이프라인)로 재설계한다.
- G7. 현재 바닐라 DOM UI를 React 컴포넌트 구조로 전환해 상태/모달/라우팅을 명시적으로 관리한다.

### Non-Goals
- N1. 사용자 계정/로그인/클라우드 동기화 (브라우저 로컬 한정).
- N2. 다중 LLM 공급자 추상화(이번 범위는 Gemini 단일). 단, 교체가 쉬운 구조는 유지.
- N3. Ollama/로컬 LLM 지원 유지. (Gemini로 대체)
- N4. 서버 측 키 보관/프록시.

## 3. 대상 사용자 & 핵심 가치

- **대상**: 카카오톡/문자 대화에서 상대 심리를 파악하고 답장을 고민하는 일반 사용자(모바일 중심).
- **가치**:
  - 프라이버시: 대화 내용이 외부(우리 서버)로 나가지 않음. LLM 호출은 사용자 본인 키로 사용자→Google 간 직접 통신.
  - 비용/운영: 운영자는 LLM 비용을 부담하지 않음. 서버는 정적 호스팅만.
  - 이식성: 어떤 정적 호스트(예: 사내 서버, Vercel/Netlify, Node 정적 서버)에도 배포 가능.

## 4. 기능 요구사항 (FR)

### 4.1 온보딩 / API 키 관리
- FR-1. 앱 최초 진입 시 키 쿠키가 없으면 **API 키 입력 모달**을 표시한다.
- FR-2. 사용자는 Google AI Studio에서 발급한 Gemini API 키를 입력한다. (키 발급 링크 안내 제공)
- FR-3. 입력한 키는 **쿠키**(`pm_gemini_key`)에 저장한다(만료 1년, `SameSite=Lax`, HTTPS에서 `Secure`).
- FR-4. 저장 전 **간단한 유효성 검증**(형식 + 1회 경량 호출로 키 동작 확인)을 수행하고, 실패 시 사유를 표시한다.
- FR-5. 설정에서 키를 **변경/삭제**할 수 있다. 삭제 시 분석 기능이 다시 잠긴다.
- FR-6. 키가 없을 때 페르소나 생성/메시지 분석 버튼은 비활성화되거나 키 입력 모달을 다시 띄운다.

### 4.2 페르소나 (기존 기능 유지)
- FR-7. 대화 기록(+선택적 "나의 이름")을 입력하면 Gemini가 상대/나 페르소나를 분석한다.
- FR-8. 분석된 페르소나는 IndexedDB에 저장되고 목록/상세/삭제가 가능하다.
- FR-9. 페르소나 상세는 기존과 동일한 8개 항목 + 나/상대 탭 구조를 유지한다.

### 4.3 메시지 분석 (기존 기능 유지)
- FR-10. 저장된 페르소나를 선택하고 받은 메시지를 입력하면 Gemini가 심리 분석 + 답변 후보 3개를 생성한다.
- FR-11. 분석 결과는 기록으로 IndexedDB에 저장된다.
- FR-12. 후보 답변 복사 기능을 유지한다.

### 4.4 기록 (기존 기능 유지)
- FR-13. 분석 기록 목록/펼치기/삭제를 유지한다.

### 4.5 상태 표시
- FR-14. 헤더의 상태 표시는 "Ollama 연결" 대신 **키 등록 여부 / 모델**을 나타낸다(예: "Gemini 준비됨" / "키 미등록").

### 4.6 React UI/UX 동기화
- FR-15. 상단 상태바는 `say-awsomely`처럼 sticky white/translucent bar, 좌측 로고+앱명, 우측 API 키 상태+언어 토글 구조를 사용한다.
- FR-16. 하단 내비게이션은 `say-awsomely`처럼 fixed bottom 3-tab bar, 활성 탭 icon scale-up, inactive slate text 스타일을 사용한다.
- FR-17. 온보딩/API 키 입력은 중앙 카드 모달 + dimmed blurred backdrop 패턴을 사용한다.
- FR-18. 생성/상세/설정 등 보조 작업은 형제 앱의 모달·다이얼로그 애니메이션과 동일한 Tailwind 토큰을 사용한다.
- FR-19. 앱별로 달라야 하는 항목은 제목, 로고/favicon, 탭명/도메인 화면, 분석 기능에 한정한다.

## 5. 데이터 & 프라이버시 요구사항
- DR-1. 페르소나/분석 기록의 모든 필드(원본 대화 포함)는 IndexedDB에만 저장한다.
- DR-2. API 키는 쿠키에만 저장한다. (사용자 명시 요구사항)
- DR-3. 우리 서버로 가는 네트워크 요청은 정적 자산 요청뿐이다. (검증: 네트워크 탭에 generativelanguage.googleapis.com 직접 호출만 존재)
- DR-4. 데이터 내보내기/가져오기(JSON)는 선택 기능으로 PLAN에 포함(스코프 허용 시).

## 6. 비기능 요구사항 (NFR)
- NFR-1. 모바일 우선, 형제 앱과 같은 React + Tailwind 디자인 토큰/레이아웃을 사용한다.
- NFR-2. TypeScript strict, 빌드 산출물은 정적(`dist/`).
- NFR-3. 첫 로드 JS는 합리적 크기(코드 스플리팅 또는 경량 SDK 사용 권장).
- NFR-4. 키/대화 데이터는 콘솔/서버 로그로 절대 유출하지 않는다.

## 7. 성공 지표 (Acceptance)
- A1. 키 미등록 신규 방문자는 온보딩 모달을 본다.
- A2. 키 등록 후 페르소나 생성·메시지 분석·기록 조회가 기존과 동일하게 동작한다.
- A3. DevTools Network에서 우리 서버로의 호출은 정적 자산뿐이고, LLM 호출은 Google 도메인으로 직접 나간다.
- A4. 새로고침/재방문 시 IndexedDB 데이터와 쿠키 키가 유지된다.
- A5. `npm run build`가 타입 에러 없이 통과하고, `npm start`로 정적 서버가 앱을 서빙한다.
- A6. 상단 상태바, 하단 탭바, 온보딩 모달, 토스트의 구조와 움직임이 `say-awsomely`와 동질적으로 보인다.

## 8. 리스크 & 완화
- R1. **키 노출(XSS)**: 쿠키/IndexedDB 모두 JS 접근 가능 → XSS 방지(출력 이스케이프 유지, 의존성 최소화). 키는 사용자 소유이며 사용자 책임 범위임을 안내.
- R2. **CORS**: Gemini REST/SDK는 브라우저 직접 호출을 지원. 미지원 변경 대비해 `gemini.ts` 한 곳에 호출을 캡슐화.
- R3. **프롬프트 회귀**: 기존 server.py 프롬프트를 그대로 이식하여 출력 품질 동일성 확보.
- R4. **모델명 변경**: 모델명을 `config.ts` 한 곳에서 관리.
